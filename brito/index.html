Concept

What is Brito?

A Claude Code plugin that turns a single Linear issue into a fully orchestrated, multi-repository implementation — without human hand-holding.

🏗️

Orchestrator, not a doer

Brito breaks work down and dispatches focused agents. Each agent receives a precise implementation plan for one repository.

📋

Linear as the brain

All state lives in Linear. Sub-issues are units of work. Comments are signals. Crashes are recoverable — just run the command again.

🤝

Human-centric handoff

Brito never marks issues as Done. It moves work to "In Review" and stops. A human decides when the work is truly complete.

Named after Saturnino de Brito — a Brazilian pioneer of urban hydraulic engineering who designed infrastructure so reliable that success was the only possible outcome. Brito the plugin engineers the environment so agents can't fail: isolated worktrees, scoped tool permissions, precise plans, and a verification loop before every push.

Agent Types · 1 / 2

Self Agents

Local execution in isolated git worktrees — running right on the machine where Brito lives.

1
Dispatch creates the sub-issue
Labeled brito-self-agent, moved to "In Progress". Contains the full per-repo implementation plan.
2
Self-Listener claims it
Posts a lock comment immediately. Creates an isolated worktree at brito/self/ISSUE_ID. No shared state with your working branch.
3
A Claude Code session executes the plan
Launched with scoped tool permissions. Makes code changes, runs tests, verifies work before committing.
4
PR created, signal posted
Session runs gh pr create. Brito detects the PR URL in output, posts completion comment to the sub-issue.
5
Worktree cleaned up
Worktree removed. Branch preserved for the PR. Sub-issue moved to "In Review".
Config
repositories:
  - name: "dash"
    url: "https://github.com/..."
    local_path: "~/dev/parsely/dash"
    agent: self   # ← that's it
Scoped tool permissions
Self-agents can: Read, Edit, Write, Grep, Glob, Bash (git, gh, npm, node), Linear MCP

Self-agents cannot: rm -rf, force push, reset --hard, branch delete
Agent Types · 2 / 2

Remote Agents

External systems pick up work from Linear. The contract: receive a plan via sub-issue, post a PR URL when done.

How it works
Dispatch creates sub-issue assigned to bot-username with label brito-remote-agent
External agent system detects the assignment via its own watcher
Reads the full implementation plan from the sub-issue description
Executes: clone → branch → implement → test → PR
Posts completion comment on the sub-issue with the PR URL
Brito's subissue-worker detects the PR URL and marks complete
Detection signals
Brito watches sub-issue comments for:
github.com/.../pull/... + "completed" or "done"
failed or blocked → marks failed, moves to In Review
agent-sandbox — example remote agent
What is agent-sandbox?
An A8C sandbox agent focused on working on the wpcom repo. It runs as an independent Claude Code session, watches Linear for sub-issues assigned to it, executes the plan autonomously, and posts the PR URL back when done.
Config
agent:
  type: issue_assignment
  assignee: "agent-sandbox"
  label: "brito-remote-agent"
The remote agent reads the sub-issue description — which contains the full, self-contained plan from Brito's dispatch template — and needs no other context to execute.
Configuration

Anatomy of brito.local.md

One global config file at ~/.claude/brito.local.md — defines all projects, repos, and agent types.

--- projects: - name: "Parse.ly Platform" # human-readable label linear_project: "https://linear.app/a8c/project/..." linear_team: "PARSELY" # issue ID prefix (e.g. PARSELY-42) repositories: - name: "dash" # used in sub-issue title: [dash] ... url: "https://github.com/Parsely/dash" local_path: "~/dev/parsely/dash" # clone path (auto-cloned if missing) agent: self # ← local agent, runs in worktree - name: "mage" url: "https://github.com/Parsely/mage" local_path: "~/dev/parsely/mage" agent: # ← remote agent type: "issue_assignment" assignee: "agent-sandbox" # Linear username to assign sub-issues label: "brito-remote-agent" # label added to trigger external watcher defaults: auto_mode: false # true = skip spec approval gate loop_interval_seconds: 120 # polling frequency for all loops max_wait_minutes: 10 # 5 iterations × 120s before timeout watcher_interval_seconds: 60 # how often /brito-watch scans watcher_max_concurrent: 2 # max parallel issues processed ---
Design Principle

Linear is the state store

No local JSON files. No SQLite. No in-memory state. Everything lives in Linear — which means everything is resumable, inspectable, and human-readable.

Comment signals Brito reads
[ag/Brito] self-agent started Lock — prevents a second session from picking up the same sub-issue
[ag/Brito] self-agent completed Completion signal — always includes the PR URL for the subissue-worker to detect
[ag/Brito] self-agent failed Failure signal — subissue-worker marks the sub-issue failed and continues
approved / lgtm / +1 Human approval of the spec — Brito continues to dispatch phase
github.com/.../pull/... PR URL in any comment — detected as completion for remote agents
Before you dispatch

The orchestration can't fix your repo.

You must make it agentic first. An agent is only as reliable as the feedback loops you give it.

🧪
Unit Testing
Agents need fast, granular feedback. Unit tests tell the agent exactly which function broke and why — without standing up the full stack. Cover edge cases, not just happy paths.
🔬
End-to-End Testing
Integration and E2E tests catch what unit tests miss: wiring errors, API contract mismatches, database migrations. An agent that can't run E2E tests is working blindfolded.
👁️
Manual Testing Hooks
Give agents a way to spin up a local environment and verify behavior visually. A dev:start script and a clear AGENTS.md tell the agent how to test its own work.
✅
Verification Loop
Before pushing, the agent must run all tests, check for lint errors, and confirm acceptance criteria from the plan. Brito uses the verification-before-completion superpowers skill for this — it refuses to push until checks pass.

The rule: if you can't tell a human how to verify that a change works in under 2 minutes, your agent can't either. Write the verification steps down in AGENTS.md — that file is the agent's operating manual.
